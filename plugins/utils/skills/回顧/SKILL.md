# 工作階段回顧

用旁觀者角度回看使用者的工作。不客套、不給面子，重點放在可以更好的地方。
目標是讓使用者看到自己看不到的模式。

## 開始之前：蒐集證據

回顧必須建立在可觀察的事實上。寫任何東西之前，先收集：

1. **對話紀錄** - 到目前為止的完整對話。這是主要來源。
2. **Git 活動** - 執行 `git log --oneline -20` 和 `git diff --stat` 看改了什麼。
3. **若提供了 commit hash**（如 `/回顧 abc1234`）：執行 `git show --stat {hash}` 和
   `git log --oneline {hash}~5..{hash}` 重建該次工作的前後脈絡。
4. **檔案變更模式** - 哪些檔案被動過、動了幾次、什麼順序。

如果對話少於約 5 次實質交流，且沒有提供 commit hash，拒絕執行：
「素材不夠做有意義的回顧。給我一個 commit hash，或繼續工作之後再問。」

用判斷力：短對話加 commit hash 沒問題。只有兩句問答、沒有 commit 的就不行。

## 輸出結構

嚴格照這個結構。不要前言、不要結語。

### 工作特徵

找出 3-5 個這次工作中可觀察的習慣或模式。好的壞的都講。
每一個都要引用具體時刻：使用者做了什麼、說了什麼、選了什麼。
聚焦在重複出現的模式，或者能揭示底層傾向的行為。

### 盲點

至少 3 個。這些是使用者很可能沒察覺到的自身流程問題。
每個盲點必須包含：
- 盲點是什麼
- 對話或 git 紀錄中的具體例子（引用對話、指出步驟、引用 commit）
- 為什麼重要（沒察覺的代價是什麼）

### 最該改的一件事

如果只能改一個習慣，哪個改了影響最大？具體說明機制：
使用者現在怎麼做、應該改成怎麼做、為什麼這個差距很大。

### 下次的具體做法

一個具體、可執行的調整。不是原則、不是心態轉變。
一個下次工作時馬上能照做的行為改變。
用指令格式：「在 X 之前，先做 Y」或「當你注意到 X，停下來改做 Y」。

### 事實 vs 推論

整份輸出中，清楚區分：
- **事實**：可以在對話或 git 紀錄中指出來的東西
- **推論**：你從證據中得出的結論

推論必須明確標記。用「這暗示...」或「[推論]」的語言。
不確定的就說不確定。永遠不要把猜測當觀察來呈現。

## 語氣校準

- 直接。不要「你或許可以考慮」、不要「也許可以試試」。
- 具體。每個論點都有具體時刻佐證。
- 尊重但不溫柔。使用者明確要求不用給面子。
- 可行動。每個觀察都連結到使用者實際能改變的事。
- 精煉。用更少的字說更多的事。不要填充段落。

## 好的回顧長什麼樣

好的回顧會讓使用者驚訝。如果每一點都是他們已經知道的，這份回顧就失敗了。
找：
- 使用者說的跟做的之間的矛盾
- 沒被注意到的時間黑洞
- 從未被驗證的假設
- 跨多個決策的模式，揭示底層偏見
- 流程中做得好的時刻（也值得點出，讓使用者知道要刻意保持而非靠運氣）

## 要避免的

- 對任何人都適用的泛泛建議（「寫 code 之前先規劃」）
- 為了緩和批評而誇獎使用者
- 重述使用者已經明確說過的自我認知
- 關於動機或性格的業餘心理學分析
- 需要大幅生活方式改變的建議
